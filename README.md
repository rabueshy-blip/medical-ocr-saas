# Medical OCR SaaS — استخراج OCR طبي هجين بدون هلوسة

Micro-SaaS يستخرج النصوص والجداول من ملفات PDF الطبية (رقمية أو ممسوحة ضوئياً) بدقة عالية،
مع بوابات برمجية صارمة تمنع أي نموذج لغوي من "اختراع" نص — موجّه للأطباء والمترجمين الطبيين.

## المبدأ الأساسي (لا يتغيّر)

**لا يُسمح لأي LLM بتوليد نص من الصفر.** كل نموذج يعمل كـ**مُصحِّح** لنص استخرجه محرك OCR
كلاسيكي أولاً (Ground Truth Anchor)، وليس كمولّد حر. كل تصحيح يُحفظ مع النص الأصلي
(audit trail)، وتُرفض برمجياً أي محاولة لتغيير رقم صريح (جرعة دواء/قيمة مخبرية) دون سند
واضح في النص الخام — راجع `medical_ocr/numeric_guard.py` والتفاصيل الكاملة في [`plan.md`](./plan.md).

## حالة المشروع الحالية (نطاق العرض التجريبي — مهم)

هذا المشروع **قيد البناء التدريجي يومياً** (راجع `plan.md` لتفاصيل كل يوم). المكوّنات
الجاهزة والمُختبرة حتى الآن:

- ✅ نموذج بيانات موحّد (`Document`/`Page`/`Block`) وتقطيع دلالي (chunking).
- ✅ موديولا DSPy: تصحيح إملائي طبي مُرسَّخ (`MedicalSpellingCorrector`)، وهيكلة جداول
  (`MedicalTableStructurer`) — كلاهما مقيّد ببوابات رفض صارمة (`is_correction_grounded`,
  `row_values_grounded`, حماية الأرقام الصريحة).
- ✅ Gold Dataset (15 عيّنة) + سكربتات تقييم/تحسين كمّي (`scripts/evaluate_gold.py`,
  `scripts/optimize_modules.py`).
- ✅ واجهة FastAPI (`/correct-spelling`, `/structure-table`, `/health`, `/docs`).
- ✅ **خط استخراج PDF كامل (`medical_ocr/ingest.py`) + واجهة Streamlit (`streamlit_app.py`)**:
  ارفع ملف PDF فعلياً واحصل على نتيجة في المتصفح — نص رقمي عبر PyMuPDF وجداول عبر pdfplumber
  للصفحات ذات طبقة نص، وOCR عبر `easyocr` للصفحات الممسوحة ضوئياً. **تبسيط متعمد** عن تصميم
  التحقق المزدوج الكامل (PaddleOCR + easyocr) الموصوف في `plan.md` — محرك OCR واحد فقط حالياً
  للصفحات الممسوحة. التصحيح بالذكاء الاصطناعي **اختياري لكل مقطع/جدول عبر زر منفصل** وليس
  تلقائياً للملف كاملاً، تحاشياً لاستهلاك حصة Gemini المجانية اليومية المحدودة دفعة واحدة.
- ✅ 71 اختباراً آلياً ناجحاً بدون أي استدعاء LM حقيقي (`tests/`).
- ✅ **أول تشغيل حقيقي ناجح ضد LM حقيقي (`scripts/run_hard_cases.py`)**: نجح النموذج في تمييز
  اسم دواء ملتبس بالسياق السريري (لا بالتشابه الحرفي فقط)، تصحيح التباس حرف/رقم من OCR، وتمييز
  خلية جدول مفقودة كـ `UNCERTAIN` بدل اختراع رقم — النتائج الكاملة في
  `scripts/hard_case_results.json`.

**غير مبني بعد:** مرحلة Triage تلقائية تختار محرك OCR الأنسب لكل صفحة بمنطق أكثر تعقيداً من
عتبة طول النص الحالية، والتحقق المزدوج (PaddleOCR + easyocr) على الصفحات الممسوحة.

**نموذج اللغة المستخدم:** Gemini عبر Google AI Studio (مجاني بالكامل، بدون بطاقة دفع) —
`gemini/gemini-3-flash-preview` افتراضياً عبر `litellm`/`dspy.LM`، بديلاً عن Anthropic الذي كان
يتطلب مفتاحاً مدفوعاً. احصل على مفتاح مجاني من https://aistudio.google.com/apikey.

**قيد الحصة المجانية (مهم قبل أي عرض حي):** المستوى المجاني من Google AI Studio يفرض حصصاً
يومية صغيرة **لكل نموذج على حدة** (بعضها بحدود 20 طلباً/يوم فقط للنماذج الأحدث، وبعضها صفر
للنماذج غير المتاحة لحسابك بعد). كل تصحيح واحد قد يستهلك عدة طلبات فعلية بسبب إعادة المحاولة
البرمجية (`dspy.Refine`, حتى 3 محاولات). جرّب `gemini-3-flash-preview` (يعمل حالياً) أولاً؛ إن
واجهت `429 RESOURCE_EXHAUSTED` انتظر إعادة تعيين الحصة (يومياً) أو جرّب نموذجاً آخر عبر
`MEDICAL_OCR_LM_MODEL` في `.env`.

## القيود البيئية

الحساب المستخدم للتطوير على macOS **ليس Administrator ولا يملك صلاحيات sudo**، لذلك:
- ممنوع Homebrew أو أي اعتمادية نظام (`tesseract-ocr`, `ghostscript`).
- كل الاعتماديات عبر `pip` فقط داخل بيئة افتراضية (`.venv`, Python 3.9.6 arm64).
- البدائل المعتمدة: `easyocr` بدل tesseract، `pdfplumber` بدل camelot/ghostscript.

## الإعداد المحلي

```bash
git clone <repo-url>
cd medical-ocr-saas
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt

cp .env.example .env
# ثم عبّئ GEMINI_API_KEY داخل .env (مجاني من https://aistudio.google.com/apikey — لا يُرفع أبداً، مُستثنى في .gitignore)
```

## تشغيل الاختبارات

```bash
.venv/bin/python -m unittest discover -s tests
```

يجب أن تنجح جميعها (71 اختباراً حالياً) دون الحاجة لأي مفتاح API — كلها بنيوية/دخان (باستثناء
اختبار واحد لاستخراج جدول حقيقي عبر pdfplumber، لا يزال بلا استدعاء LM).

## تشغيل واجهة Streamlit (العرض التجريبي الأسهل — ارفع PDF وشاهد النتيجة)

```bash
.venv/bin/streamlit run streamlit_app.py
```

يفتح المتصفح تلقائياً على `http://localhost:8501`. ارفع ملف PDF فتظهر نتيجة الاستخراج فوراً
(مجانية، بلا LM) لكل صفحة/فقرة/جدول. **زر التصحيح بالذكاء الاصطناعي منفصل لكل مقطع** — اضغطه
فقط عند الحاجة الفعلية، لتفادي استهلاك حصة Gemini المجانية اليومية المحدودة على ملف واحد.
يعمل الاستخراج نفسه (بدون تصحيح) حتى بلا `GEMINI_API_KEY` في `.env`.

## تشغيل واجهة API محلياً (للعرض التجريبي)

```bash
.venv/bin/python -m uvicorn medical_ocr.api.app:app --reload --port 8000
```

- `GET /health` — فحص الحالة (يعمل حتى بدون مفتاح API، ويُظهر `lm_configured: false` إن غاب).
- `GET /docs` — واجهة Swagger التفاعلية؛ أسهل طريقة لعرض النقاط للأطباء/المترجمين دون كتابة كود.
- `POST /correct-spelling` — يتطلب `GEMINI_API_KEY` فعلي، وإلا يرجع `503` برسالة عربية واضحة.
- `POST /structure-table` — نفس الشرط.

مثال طلب (بعد توفر مفتاح API فعلي):

```bash
curl -X POST http://127.0.0.1:8000/correct-spelling \
  -H "Content-Type: application/json" \
  -d '{"raw_text": "المريض يتناول ميتفورمين 850 ملغ مرتين يومياً"}'
```

## الأمان والخصوصية (مهم قبل أي عرض تجريبي فعلي)

- **لا ترفع بيانات مرضى حقيقية** إلى هذا المستودع أو إلى أي طلب API تجريبي — استخدم بيانات
  مصطنعة أو مُجهَّلة فقط (كل عيّنات `data/gold/` حالياً مصطنعة بالكامل لهذا الغرض).
- `.env` مُستثنى من git (`.gitignore`)، ولا يحتوي المستودع أي مفتاح API حقيقي — تم التحقق يدوياً.
- المستودع **خاص (private)** على GitHub لحماية الكود قبل مرحلة الإطلاق.
- راجع `LICENSE` — كل الحقوق محفوظة، لا ترخيص عام للاستخدام أو التوزيع.
